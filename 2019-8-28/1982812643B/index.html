针对聊项目的时候的问题"你这些组件是怎么做的?"的一些回答方向

项目背景:

一个中小型项目的重构, pc 端, 利用 nw.js 打包的一个桌面应用。要从 jq 的多页面重构为 vue 的单页面。

这个重构项目的前期工作都是我做的，包括环境配置，旧项目的功能，相似部分抽取，新模块的组件划分、组织（目录结构），外部链接页面的接入与交互，和之后的实现等。

思路

在面试中谈到了这个项目，被问到“你这些组件是怎么做的？”，但是我答的并不能让面试官满意。

我是从这些方面答的：

  1. 从旧项目中怎么把页面中一些重复部分提取出来
  2. 提取出来的东西怎么划分组件
  3. 这些组件的分类, 和他们是通过什么交互的
  4. 这样子划分的好处

MeatHill 老师的回答

在请教 MeatHill 老师之后, 他给出的回答方向是

  1. 组件化的使用率, 即可复用性, 是否真的节省了成本
  2. 这些组件如何应对不同需求的变化
  3. 这些组件发现了什么问题, 如何解决?

    关于这点, 他举了个例子, “比如我厂的表格组件，使用的是 column/cell 结构，但是表格数据有非常高的基于 row 聚合的特性”

  4. 这些组件成熟后, 是否试过仓库化, 成为一个独立的依赖, 每个组件是否有一套单元测试

总结

我在团队中的角色是执行者, 我的回答也是站在执行者“怎么做”的方向去考虑。

换一个角度，站在整个团队去考虑的话，应该从复用性，即插即用，应对需求，和仓库化的方向去考虑，另外需要考虑的是单元测试跟如何去解决问题。

针对目前在这个项目所设计的组件，我考虑到了复用性，通过将部分公用功能划分为公共组件，并且在功能上也做到了独立。也有被其他项目引用的案例，但是目前的还需要考虑样式跟图片的问题。

  1. 没有默认样式，每次其他项目引用都需要修改掉样式跟图片
  2. 没有考虑到需求变更的问题，没有往更多方向的状态去考虑

在一部分的组件上考虑到了应对需求的变化，但更多是考虑到业务方面更大方向的需求变更，所以更多的是在组件组织跟数据组织方面的解耦跟冗余。缩小到组件层面，并没有过多考虑到组件的各种形态的可能性。仅留下了空余用于日后补充。

因此，以后设计组件还需要考虑的方向：

  1. 表现形式的不一致，
  2. 数据的不一致（比如说有跟没有图片）
  3. 应对需求的能力（站在组件的层面）
  4. 这些组件如何测试

最后要感谢 MeatHill 老师！
